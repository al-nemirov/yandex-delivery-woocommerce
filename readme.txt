Полная интеграция WooCommerce с Яндекс Доставкой: расчёт стоимости, выбор ПВЗ на карте, выгрузка заказов, печать ярлыков, синхронизация статусов, МойСклад.

== Description ==

Плагин для интеграции WooCommerce с Яндекс Доставкой через официальный B2B API.

= Способы доставки =

* **ПВЗ (самовывоз)** — клиент выбирает пункт выдачи на карте Яндекс
* **ПВЗ с наложенным платежом** — оплата при получении в пункте выдачи
* **Курьер до двери** — доставка по адресу клиента
* **Курьер с наложенным платежом** — оплата курьеру при получении

= Расчёт стоимости =

* Динамический расчёт через API Яндекс Доставки (вес, габариты, маршрут)
* Единая формула расчёта для корзины и заявки (yd_assessed_price_minor_units + yd_calculate_package_dims)
* Передача platform_station_id для склада и ПВЗ — корректный расчёт по тарифу договора
* Фиксированная наценка в рублях поверх цены API
* Процентная наценка поверх цены API
* Фоллбек на фиксированную стоимость при ошибке API
* Отображение срока доставки в днях
* Кэширование результата (ПВЗ и ПВЗ с нал.платежом не дублируют API-вызов)
* Фильтрация по мин/макс весу отправления

= Выбор ПВЗ на чекауте =

* Интерактивная карта Яндекс с кластеризацией точек
* Список ПВЗ с адресами и графиком работы
* Поиск по адресу внутри модалки
* Цена не показывается пока ПВЗ не выбран — заглушка «выберите пункт выдачи»
* Сброс ПВЗ при смене города — cookies и UI автоматически очищаются
* Сохранение выбора в cookies для устойчивости при обновлении чекаута

= Подсказки адресов (Dadata) =

* Автоподсказки города при вводе
* Автоподсказки адреса для курьерской доставки
* Настройка API-ключа Dadata в отдельном меню

= Выгрузка заказов =

* Создание заявки на доставку в Яндекс через API
* Авто-отправка при переходе в выбранный статус
* Ручная отправка кнопкой из карточки заказа
* Повторная отправка при ошибке (retry)
* Подтверждение заказа в API
* Настраиваемый префикс номера заказа

= Печать =

* Генерация и скачивание ярлыков (PDF)
* Генерация и скачивание актов передачи
* Авто-генерация акта после выгрузки заказа

= Синхронизация статусов =

* Крон-задача каждые 2 часа — опрос API Яндекс Доставки
* Автоматическое обновление статуса WooCommerce-заказа
* Авто-завершение заказа при вручении клиенту
* Полная история статусов в карточке заказа

= Интеграция с МойСклад =

* Автоматическая выгрузка заказов в МойСклад при смене статуса
* Ручная выгрузка кнопкой из карточки заказа
* Создание контрагентов и заказов покупателей
* Настройка организации и товара по умолчанию

= Автообновление =

* Плагин автоматически проверяет новые версии на GitHub каждые 12 часов
* Обновление через стандартный механизм WordPress (Плагины → Обновления)
* Не требует сторонних плагинов

= Администрирование =

* Метабокс в карточке заказа: трекинг, ярлыки, акты, ошибки
* Настройка пункта приёма с автокомплитом из API (12000+ точек)
* Проверка валидности API-токена
* Тестовый режим (скрывает методы доставки, не затрагивает боевые заказы)
* Совместимость с HPOS (High-Performance Order Storage)
* Совместимость с PHP 8.1+

= Чекаут =

* Поддержка классического и блочного чекаута WooCommerce
* Валидация: обязательный выбор ПВЗ для самовывоза
* Автозаполнение billing-полей для ПВЗ (адрес, индекс)
* Кастомные CSS-стили для чекаута

= Вес и габариты =

* Единая функция расчёта yd_calculate_package_dims() — и для корзины, и для заявки
* Автоматический расчёт из свойств товаров WooCommerce
* Конвертация единиц (кг/г, см/м/мм) через yd_unit_coefficients()
* Укладка стопкой: max(длина) × max(ширина) × сумма(высот)
* Габариты по умолчанию: к каждому товару или ко всему отправлению
* minWeight применяется к отправлению, а не к каждому товару

= Безопасность =

* Nonce-проверка на всех AJAX-обработчиках
* Capability-проверки на админских эндпоинтах
* XSS-экранирование в виджете ПВЗ и админ-панели
* Логирование API только при WP_DEBUG, адреса маскируются для защиты ПДн
* Страна по умолчанию из настроек WooCommerce (фильтр yd_default_billing_country)

== Installation ==

1. Загрузите папку `yandex-dostavka` в `wp-content/plugins/`
2. Активируйте плагин через меню «Плагины»
3. Перейдите в **WooCommerce → ЯД — Настройки API** — введите ключ Dadata и Яндекс Карт
4. Перейдите в **WooCommerce → Доставка → Зоны** → добавьте метод «Яндекс Доставка — ПВЗ»
5. Введите OAuth-токен из личного кабинета Яндекс Доставки
6. Выберите пункт приёма заказов (начните вводить город)
7. Настройте наценки и автоматическую отправку

Обновления приходят автоматически через GitHub Releases.

== Frequently Asked Questions ==

= Где взять OAuth-токен? =

В личном кабинете Яндекс Доставки → раздел «Интеграция» → API-токен.

= Где взять ключ Dadata? =

Зарегистрируйтесь на dadata.ru → Личный кабинет → API-ключи.

= Как работает наложенный платёж? =

Добавьте метод «Яндекс Доставка — ПВЗ (наложенный платёж)». Клиент оплачивает при получении. Сумма = товары с НДС + доставка с НДС.

= Как настроить МойСклад? =

WooCommerce → МойСклад Яндекс — введите логин/пароль, UUID организации и товара по умолчанию.

= Как работает автообновление? =

Плагин проверяет GitHub Releases каждые 12 часов. Если есть новый тег (например v2.3.0), WordPress покажет обновление в стандартном интерфейсе. Для публикации: `gh release create v2.3.0 --title "v2.3.0" --notes "описание"`.

== Known Limitations ==

= Размер посылки (S/M/L/XL) не выбирается автоматически =

API Яндекс.Доставки не поддерживает поле для автоматического выбора тарифного размера посылки. Подтверждено ответом техподдержки ЯД (апрель 2026): **«Нет параметра в API, который бы самостоятельно подтягивал габариты. Вам нужно их самому передавать».**

Плагин корректно передаёт `weight_gross` и `dx/dy/dz` в `places[].physical_dims` при создании заказа (`/api/b2b/platform/request/create`), но в курьерском приложении Яндекса при этом активируется вкладка **«Ввести»** (пользовательские габариты), а не одна из предустановленных размерных кнопок. Как следствие — на PDF-этикетке может печататься вес `0`, если оператор/курьер не нажмёт размерную кнопку вручную в аппе.

Workaround со стороны магазина: **нет.** Это ограничение серверной части Яндекс.Доставки. Ожидаем расширения API.

**Feature request к команде API Яндекс.Доставки:** добавить в `places[].physical_dims` поле типа `size_code` (`s` / `m` / `l` / `xl` / `xxl` / `xxl_plus` / `kgt`), чтобы внешние системы могли явно передавать тарифный размер. Либо сделать автоопределение по весу на стороне сервера при приёме заказа через `/api/b2b/platform/request/create`. Keywords для поиска коллегам из Яндекс.Доставки: `yandex delivery api`, `places physical_dims size_label`, `pricing-calculator size code`, `b2b-authproxy.taxi.yandex.net`, `Ввести вместо S M L XL XXL`.

Публичный GitHub Issue для обсуждения: https://github.com/al-nemirov/yandex-delivery-woocommerce/issues — если читаете это как разработчик из Яндекса и хотите поделиться ETA по фичреквесту, будем благодарны за комментарий.

Таблица соответствия весов (для справки оператору / курьеру):

* **S** — до 2 кг
* **M** — до 5 кг
* **L** — до 12 кг
* **XL** — до 15 кг
* **XXL** — до 20 кг
* **XXL+** — до 30 кг
* **КГТ** — до 200 кг

= BETA-тарифы требуют подписки в ЛК =

Методы `yd_express` (тариф `express`) и `yd_same_day` (тариф `same_day`) требуют активной подписки на соответствующий тариф в ЛК Яндекс.Доставки. Если подписка не подключена, метод **автоматически скрывается** на чекауте (see v2.16.1-beta) — клиент не видит недоступный способ доставки, в wp-admin появляется баннер с текстом ошибки API.

= WP-Cron зависит от трафика =

Twice-daily-крон обновления справочника ПВЗ срабатывает только при посещениях сайта — это особенность WordPress, а не плагина. Для сайтов с низким трафиком рекомендуется настроить системный cron:

`0 3 * * * curl -s https://example.com/wp-cron.php?doing_wp_cron >/dev/null`

Защита от устаревших данных — lazy-refresh при фронт-запросе, если справочник старше 3 дней (настраивается фильтром `yd_pvz_stale_after_seconds`).

== Changelog ==

= 2.16.2-beta =

*Changelog теперь читается из readme.txt:*

Раньше в WP-админке на экране «Детали обновления» (модалка после клика по «Посмотреть детали версии») отображалось тело GitHub Release — но GitHub автогенерирует его как пустышку вида `Full Changelog: compare/v2.13.0...v2.16.1-beta`. Поэтому даже после обновления плагина пользователь не видел, что именно изменилось.

Решение: `yd_github_plugin_info()` теперь подтягивает секцию `== Changelog ==` из `readme.txt` на GitHub **по конкретному тегу** (`raw.githubusercontent.com/…/<tag>/readme.txt`), парсит её в HTML и отдаёт в `plugins_api → sections.changelog`. Фолбэк на ветку `main`, если тег недоступен. Если и она недоступна — фолбэк на старое поведение (body релиза).

Мини-парсер WordPress.org readme-формата встроен в плагин — без зависимостей. Поддерживает: `= версия =`, `*курсив*`, `**жирный**`, `` `код` ``, маркированные и нумерованные списки. Кэш через transient `yd_github_readme_changelog` на 6 часов; сбрасывается по `?force-check=1` вместе с `yd_github_release`.

Эффект: теперь достаточно обновлять readme.txt в репо (одним местом), и все клиенты при следующем апдейте сразу увидят подробный changelog в WP-админке.

= 2.16.1-beta =

*Добавлено — защита BETA-тарифов от "Экспресс за 350 ₽":*

Проблема: если админ включил `yd_express` или `yd_same_day` в зоне доставки, но забыл подключить соответствующий тариф в ЛК Яндекс.Доставки — плагин молча падал в `fixed_cost` (по умолчанию 350 ₽) и продавал клиенту премиум-доставку по неправильной цене.

Решение — три слоя защиты:

1. **Runtime hide.** Для методов с `is_beta=true` при ЛЮБОЙ ошибке API (нет подписки, город не обслуживается, любая причина) `calculate_shipping()` возвращает `false` — rate просто не добавляется на чекаут. Никакого `fixed_cost` fallback для BETA.
2. **Failure cache.** Ошибка сохраняется в transient `yd_beta_fail_<method_id>` на 1 час. Следующие чекауты в течение часа не дёргают API — сразу прячут метод. Ноль нагрузки на API, ноль задержек пользователю.
3. **Admin signal.** Опция `yd_beta_tariff_failures` хранит последнюю ошибку по каждому методу. В wp-admin показывается dismissable-баннер: тариф, текст ошибки, время, ссылка на ЛК Яндекс, кнопка "Сбросить флаг" (удаляет и опцию, и transient). Также — предупреждение прямо в форме настроек метода (WC → Shipping zones).

Успешный расчёт цены автоматически сбрасывает transient с ошибкой — как только подписка включена в ЛК, метод появится на чекауте в течение часа без участия админа.

= 2.16.0-beta =

*Новые способы доставки (BETA):*

* **yd_express** — тариф API `express`, доставка за 1-2 часа
* **yd_same_day** — тариф API `same_day`, доставка в день заказа

Оба метода требуют подключения соответствующей подписки в ЛК Яндекс.Доставки. Появляются в WooCommerce → Shipping zones → Add shipping method как «Яндекс Доставка — Экспресс (BETA)» и «Яндекс Доставка — В день заказа (BETA)».

*Рефакторинг:*

* `WC_YD_Parent_Method::$tariff` — явное свойство тарифа вместо ветвления по булевому `self_type`. Дочерний класс задаёт `$this->tariff = 'express'` напрямую. Старая логика (`self_type ? 'self_pickup' : 'time_interval'`) сохранена как fallback для классов без явного тарифа — обратная совместимость полная.
* Два центральных хелпера: `yd_all_method_ids()` и `yd_pvz_method_ids()` (оба фильтруются). Заменили 4 хардкодных массива `array('yd_self', 'yd_self_after', ...)`, разбросанных по файлу. Кастомные subclass-ы извне автоматически получают: поиск токена, обновление ПВЗ, мета-боксы, подстановку ПВЗ-кода — без правки ядра.
* `last_mile_policy` в `request/create` оставлено `time_interval` для всех курьерских — API Яндекса принимает только `self_pickup | time_interval` в этом поле; скорость курьера определяет сам тариф.

= 2.15.4 =

*Самолечение справочника ПВЗ — больше не требует ручного пересохранения настроек:*

Проблема: таблица `yd_reception_points` протухала (или оставалась пустой после обновления плагина через FTP) — и админу приходилось заходить в WC → Shipping zones и нажимать «Сохранить» на методе, чтобы ПВЗ обновились.

Решение — четыре слоя автообновления:

1. **Крон самоперерегистрируется на каждый `init`** (не только на `register_activation_hook`). Если плагин обновили через FTP без деактивации — cron-хук воскреснет на следующей загрузке любой страницы.
2. **Twice-daily cron `yd_update_data_event`** (2 раза в сутки) — как и раньше, вызывает `yd_add_reception_points()`.
3. **Lazy-refresh при фронт-запросе.** Если таблица пустая ИЛИ последний рефреш был > 3 дней назад — следующий веб-запрос (не AJAX, не cron) триггерит одноразовое обновление до ответа клиенту. Throttling через transient на 1 час — одновременные визиты не запустят API дважды.
4. **Метка `yd_pvz_last_refresh`** обновляется в `yd_add_reception_points()` после любой успешной загрузки (крон, ручное сохранение, lazy). Сбрасывает таймер staleness.

Пороги настраиваются:

```php
add_filter('yd_pvz_stale_after_seconds', fn() => DAY_IN_SECONDS); // проверять раз в сутки
add_filter('yd_pvz_query_limit', fn() => 5000);                   // больше ПВЗ на город
```

Для разового принудительного обновления: `delete_option('yd_pvz_last_refresh')`.

= 2.15.2 =

*Три точечных фикса по аудиту логов:*

* **Фамилия получателя.** Если в заказе пусто — раньше подставлялось `«—»` (Unicode em-dash), что выглядело уродливо на этикетках и актах. Стало: дублируется имя (фильтр `yd_recipient_last_name_fallback`). Для полностью пустого ФИО — `.`.
* **Дубли pricing-calculator.** В логах видели два идентичных POST /api/b2b/platform/pricing-calculator в одну и ту же секунду — это параллельные AJAX от WooCommerce checkout (update_order_review + set_address). `$GLOBALS['yd_pricing_cache']` помогает внутри одного PHP-процесса, но не между параллельными. Добавлен transient-кэш на 30 секунд по MD5 тела запроса прямо в `Yandex_Delivery_API::calculate_price()` — успешные ответы кэшируются, ошибки не кэшируются (не залипнем на временном сбое).
* **Крон пропускает мёртвые заказы.** Было: `yd_sync_order_statuses()` тянул `get_request_info` для всех заказов со статусами processing/on-hold/pending каждые 2 часа, включая давно отменённые Яндексом (`SHOP_CANCELLED`, `RETURNED`…). Стало: если `yd_last_status_code` в терминальном списке (`DELIVERED`/`DELIVERED_FINISH`/`CANCELLED`/`SHOP_CANCELLED`/`RETURNED`/`RETURNED_FINISH`/`RETURNED_TO_SHOP`/`RETURNED_DELIVERED`/`FAILED`) — skip. Список расширяется фильтром `yd_terminal_status_codes`.

= 2.15.1 =

* ПВЗ-поиск по городу: убран 3-символьный fallback — из-за него `«Калининград»` находил «Калугу» (по префиксу «Кал»), а `«Псков»` мог подхватить «Псковская улица» в Москве. Теперь — строгая 3-ступенчатая стратегия от `LOWER(city) = 'псков'` к `city LIKE 'псков%'` и `city LIKE '%псков%'`, строго по колонке `city` (из `address.locality`), а не по `name` (полному адресу).

= 2.15.0 =

*Крупный фикс загрузки ПВЗ и весов:*

* **Загрузка ПВЗ.** Убран `available_for_dropoff=true` из `pickup-points/list` — этот фильтр подтягивает только «куда магазин везёт посылки», а целые города (Псков, Калининград) туда не попадают, хотя Яндекс туда доставляет. Сейчас используется `{type: 'pickup_point'}` без дополнительных фильтров — ~30k точек по РФ.
* **Item weight.** В `create_request` появился двойной фикс веса: (1) `Yandex_Delivery_API::fix_items_weight()` пересчитывает `items[].weight` по SKU из WooCommerce (через `bxbGetWeight()` + `yd_unit_coefficients()`), дефолт 500 г; (2) в основном билдере заказа `physical_dims.weight_gross` считается из `bxbGetWeight( $product, $variation_id )` с коэффициентом единиц (1000 для кг, 1 для г).
* **Парсинг `payment_methods` из `pickup-points/list`.** Яндекс возвращает разные форматы: `['postpay','card_on_receipt']`, `'card_on_receipt'` строкой, `[{type:'postpay'}]` объектами. Нормализуем в единый массив строк. Флаг `cash_allowed = 1` если ПВЗ принимает `card_on_receipt`. Если API не вернул поле — считаем `1` (не скрываем точки). Fallback на устаревшие поля `cash`/`payment_on_delivery`.
